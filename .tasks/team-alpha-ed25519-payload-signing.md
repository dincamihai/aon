# Sub B — Replace HMAC envelope with Ed25519 per-role signing

Part of [`team-alpha-crypto-identity-integrity.md`](team-alpha-crypto-identity-integrity.md).
**Blocks on Sub A** ([`nsc-jwt-migration.md`](nsc-jwt-migration.md)).
Reason: Sub A's `nsc add user` produces the Ed25519 nkey seed that this
card uses as the signing private key. Same key, two purposes (NATS
auth + envelope signing). Do not generate a parallel Ed25519 keystore.

**Deferred** — depends on `nsc-jwt-migration.md`. Ed25519 keypair lifecycle
reuses nkey seeds generated by `nsc add user`.

Supersedes the symmetric-secret model from
`team-alpha-hmac-payload-signing.md` once landed. HMAC was MVP for tamper
evidence; Ed25519 adds per-role identity proof (cannot forge without
private key) and removes the shared-secret blast radius.

## Why replace, not layer

- Single signing system → one rotation, one verification path, one mode
  env. Layering doubles key management with no offsetting gain — Ed25519
  on its own already covers tamper + identity.
- Public-key cryptography means any compromised role leaks only that
  role's identity, not cluster-wide signing capability.
- nkey infra (post-JWT) already produces Ed25519 keypairs; piggyback on
  it, don't run a parallel scheme.

## Scope

1. **Key source**: each role's Ed25519 keypair derived from its nkey seed
   (NSC creds file). Private key stays on role's host, never leaves.
2. **Public key map**: `agents/<role>.json` gains a `pubkey` field
   (Ed25519 hex). Source of truth in git, same place as skill cards.
   Loader caches map at MCP server startup; reload on SIGHUP.
3. **`crypto.py` v2**:
   - Add `cryptography` (or `pynacl`) to mcp-server deps.
   - `sign_envelope_ed25519(payload, role) -> dict` — same envelope
     shape (`v`, `by`, `ts`, `nonce`, `payload`, `sig`) but `sig` =
     Ed25519 hex over canonical JSON of the rest.
   - `verify_envelope_ed25519(env, *, expected_role=None)` — looks up
     `env.by`'s public key from cards, verifies. Same replay/stale
     guards.
   - Bump `v` to `2`. Verifier accepts both `v=1` (HMAC) and `v=2`
     (Ed25519) during transition window; flips to v2-only after rollout.
4. **Mode env extends**: `TEAM_ALPHA_HMAC_MODE` renamed
   `TEAM_ALPHA_SIG_MODE` ∈ {off, warn-hmac, strict-hmac, warn-ed25519,
   strict-ed25519}. Migration period only — collapse to {off, warn,
   strict} after HMAC retired.
5. **`aon` CLI**: replace `aon hmac` with `aon sig` subcommand:
   - `aon sig genkey` — derive Ed25519 keypair from nkey seed; write
     `agents/<role>.json` pubkey field; private stays in NSC creds.
   - `aon sig status` — show key, mode, public-key map.
   - `aon sig mode <off|warn|strict>` — persist mode.
6. **Drop HMAC code** once cluster fully on v2 envelopes (separate
   cleanup slice).

## Migration path

- HMAC ships first (current state, S1+S2+S3 done).
- JWT migration lands → role nkey creds available everywhere.
- Operator runs `aon sig genkey` per role; PRs the pubkey additions
  to `agents/<role>.json`; cluster reloads.
- Flip `TEAM_ALPHA_SIG_MODE=warn-ed25519`; soak; flip
  `strict-ed25519`. Drops HMAC verifier in next release.

## Acceptance

- [ ] `agents/<role>.json` schema gains required `pubkey` (Ed25519 hex).
- [ ] `crypto.py` ships v2 sign/verify backed by Ed25519, with public-key
      map loader.
- [ ] All A2A `tasks/send`, status, message, DM, broadcast envelopes
      sign with sender role's private key.
- [ ] Receivers verify with sender's public key; reject under
      `strict-ed25519` if missing/bad sig.
- [ ] Forge test: agent A constructs envelope with `by: B` and any sig
      → verifier rejects (sig fails B's pubkey check). HMAC could not
      catch this; Ed25519 does.
- [ ] Tamper test, replay test, stale test all pass as in HMAC version.
- [ ] Public-key rotation: rotate one role's keypair, old envelopes
      verify only within retention window using cached old pubkey.
- [ ] Perf: <2ms sign+verify p99 (Ed25519 ~30µs each).
- [ ] HMAC verifier removed in cleanup slice.

## Non-goals

- Multi-operator federation (would need a CA or SPIFFE — separate
  ADR if ever needed).
- Encrypting payloads (TLS covers transit; storage encryption
  orthogonal).
